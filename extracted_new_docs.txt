Verificar firma HMAC-SHA256 del header X-Hub-Signature-256.

GPT-4o responde con opciones de horario disponibles.

ℹ️  DECISIONES DEFINITIVAS DE ARQUITECTURA
Auth: Clerk (NO NextAuth.js). Un solo proveedor para toda la plataforma.
Database: Neon (PostgreSQL serverless) + Prisma ORM.
Almacenamiento wizard: localStorage para borrador temporal. PostgreSQL para persistencia real.
Upload: UploadThing. Alternativa: Cloudinary si se requieren transformaciones de imagen en servidor.
Cache/Queue: Upstash Redis (compatible con Vercel Edge Runtime y serverless). NO usar ioredis.
Real-time: React Query polling 30s. NO Socket.io (incompatible con Vercel serverless).
Cal.com: self-hosted en Railway. TODA operación de disponibilidad y reserva pasa por Cal.com API. Nunca escritura directa a BD.
IA: OpenAI GPT-4o + Vercel AI SDK (streaming). LangChain.js para orquestación.
Email: Resend + React Email.
Monitoreo: Sentry (errores) + PostHog (analytics de producto).

// lib/buildSystemPrompt.ts
async function buildSystemPrompt(restaurantId: string): Promise<string> {
  // Cachear en Redis con TTL 10 minutos
  const cached = await redis.get(`restaurant:${restaurantId}:system-prompt`);
  if (cached) return cached as string;

  const restaurant = await db.restaurant.findUnique({
    where: { id: restaurantId },
    include: { schedules: true, tableTypes: true }
  });

  const prompt = `Eres el asistente virtual de ${restaurant.nombre}, un ${restaurant.tipo}
ubicado en ${restaurant.ciudad}.`,

PERSONALIDAD: ${restaurant.agentPersonality ?? 'amigable y profesional'}

HORARIOS: ${formatSchedules(restaurant.schedules)}

MESAS DISPONIBLES:
${restaurant.tableTypes.map(t => `- ${t.nombre}: ${t.minCapacity}-${t.maxCapacity} personas`).join('\n')}

REGLAS ESTRICTAS:
- Responde en español (o el idioma del cliente si escribe en otro idioma).
- NUNCA inventes disponibilidad. USA SOLO la herramienta check_availability.
- Para grupos de +${restaurant.maxPartySize}: solicitar contacto directo.
- USA eventTypeId: ${restaurant.defaultEventTypeId} para consultar disponibilidad.`,

  await redis.setex(`restaurant:${restaurantId}:system-prompt`, 600, prompt);
  return prompt;
}

ℹ️  DECISIÓN DEFINITIVA: sessionStorage (NO localStorage)
sessionStorage es la elección correcta para el flujo de reserva porque:
Una reserva NO debe 'recordarse' al reabrir el navegador (contiene datos personales sensibles).

const pusher = new Pusher({
  appId: process.env.PUSHER_APP_ID!,
  key: process.env.PUSHER_KEY!,
  secret: process.env.PUSHER_SECRET!,
  cluster: process.env.PUSHER_CLUSTER!,
});

export async function POST(request: Request) {
  const { userId, orgId } = auth();
  if (!userId) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  const body = await request.formData();
  const socketId = body.get('socket_id') as string;
  const channel = body.get('channel_name') as string;

  // Verificar que el canal solicitado corresponde al restaurante del usuario
  const expectedChannel = `private-restaurant-${orgId}`;
  if (channel !== expectedChannel) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }

  const authResponse = pusher.authorizeChannel(socketId, channel);
  return NextResponse.json(authResponse);
}
