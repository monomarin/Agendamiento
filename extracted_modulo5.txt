📋 Resumen Ejecutivo
El Módulo 5 de Antigravity es la Capa de Integración y API Pública: la interfaz que permite que sistemas externos (POS, Google Maps, Zapier) consuman y alimenten datos de la plataforma. Es el módulo con mayor exposición al exterior y, por tanto, el de mayor riesgo de seguridad de toda la plataforma.

Este documento analiza el prompt técnico original del Módulo 5 e identifica errores críticos de arquitectura, bugs de código, inconsistencias con los módulos anteriores y decisiones de diseño que, si se implementan tal como están, generarían brechas de seguridad, inconsistencias de datos con Cal.com o una experiencia degradada para los desarrolladores que consuman la API. El objetivo es que el equipo inicie la implementación con una base sólida y sin retrabajo costoso.

⚠️ PROBLEMA CRÍTICO: El endpoint POST /api/v1/bookings bypasea Cal.com
El servicio createBookingService() del documento crea la reserva directamente en PostgreSQL via Prisma. Este es exactamente el mismo anti-patrón ya identificado y corregido en el Módulo 3 (Sección 1.3). CONSECUENCIA: La reserva existe en Antigravity pero NO en Cal.com. El slot queda disponible en el motor de disponibilidad → dos clientes pueden reservar la misma mesa simultáneamente. El restaurante verá el slot libre en Cal.com y lo podrá bloquear manualmente, generando un doble-booking. SOLUCIÓN OBLIGATORIA: Igual que en Módulos 2, 3 y 4, toda creación de reserva debe seguir el flujo: 1. Verificar disponibilidad → Cal.com GET /api/v2/slots 2. Crear reserva → Cal.com POST /api/v2/bookings 3. Cal.com dispara webhook BOOKING_CREATED → Antigravity lo recibe y escribe en BD 4. Responder al cliente externo con { bookingId, confirmationCode } El createBookingService() del Módulo 5 DEBE ser el mismo servicio ya implementado en el Módulo 4 (lib/calcom.ts createBooking()), no uno nuevo.
⚠️ ERROR CRÍTICO DE SEGURIDAD: El rate limit usa request.ip en lugar del API Key
En el middleware de autenticación, el rate limiting se aplica sobre la IP del cliente: const ip = request.ip || '127.0.0.1'; const { success } = await ratelimit.limit(ip); PROBLEMA 1: Un atacante puede cambiar de IP (VPN, proxies) para evitar el límite. PROBLEMA 2: Múltiples restaurantes legítimos detrás del mismo NAT corporativo compartirán el mismo límite y se bloquearán entre sí. PROBLEMA 3: Los límites por plan (Free: 100/h, Pro: 1000/h) no se pueden aplicar por IP; requieren conocer el plan del restaurante, que está en keyRecord. SOLUCIÓN: Aplicar rate limit sobre el API Key o restaurantId, que ya está validado en ese punto del middleware: const { success, limit, reset, remaining } = await ratelimit.limit(keyRecord.restaurantId); Usar un límite diferente por plan: crear instancias separadas de Ratelimit para cada plan (FREE_LIMITER, PRO_LIMITER) y seleccionar según keyRecord.restaurant.plan.
⚠️ PROBLEMA DE SEGURIDAD: isActive no es suficiente para invalidar una clave comprometida
El middleware verifica únicamente keyRecord.isActive. Sin embargo, el esquema del documento no incluye campos de expiración ni rotación de claves. ESCENARIOS NO CUBIERTOS: • Un API Key filtrado a internet no tiene forma de expirar automáticamente (solo revocar manualmente). • Las claves de test (res_test_...) deberían bloquearse automáticamente en el entorno de producción. • Si un desarrollador externo abandona el proyecto, su clave seguirá activa indefinidamente. SOLUCIÓN: Agregar campos al schema de ApiKey: model ApiKey {   expiresAt   DateTime?   // null = sin expiración   lastUsedAt  DateTime?   // para detectar claves inactivas   environment String      @default('live') // 'live' | 'test' } En el middleware, verificar: if (keyRecord.expiresAt && keyRecord.expiresAt < new Date()) → 401 'API Key expirada'. Las claves test solo funcionan en el entorno de staging (variable de entorno NODE_ENV).
⚠️ PROBLEMA CRÍTICO: triggerWebhook() se llama de forma síncrona dentro del request
La función triggerWebhook() usa axios.post() directamente con un timeout de 30 segundos. Si el endpoint del restaurante está caído o lento, el request del cliente externo esperará hasta 30 segundos antes de recibir respuesta. CONSECUENCIA: Una reserva creada por el POS del restaurante puede tardar 30 segundos en responder porque el webhook de Slack del restaurante está caído. Esto es inaceptable en producción. SOLUCIÓN: Encolar el disparo del webhook en BullMQ (ya disponible del Módulo 2) y responder inmediatamente: // Responder al cliente externo inmediatamente const booking = await createBookingViaCalCom(data); // Encolar el webhook sin await — no bloquear webhookQueue.add('trigger-webhook', { event: 'booking.created', payload: booking }); return NextResponse.json({ success: true, data: booking }, { status: 201 }); El worker de BullMQ maneja los reintentos con backoff exponencial, el historial de entregas y los timeouts de forma independiente.
⚠️ PROBLEMA DE COMPATIBILIDAD: axios no funciona en Vercel Edge Runtime
El documento usa import axios from 'axios' en lib/webhooks.ts para disparar los webhooks. Las API Routes de Next.js en Vercel pueden ejecutarse en Edge Runtime, donde axios no está disponible (es un módulo de Node.js). CONSECUENCIA: El disparo de webhooks fallará silenciosamente en producción si la ruta se configura como Edge Function. En local (Node.js) funciona sin problema. SOLUCIÓN: Reemplazar axios por la fetch API nativa que sí es compatible con Edge Runtime: // INCORRECTO await axios.post(webhook.url, body, { headers, timeout: 30000 });  // CORRECTO — compatible con Edge y Node.js const controller = new AbortController(); const timeoutId = setTimeout(() => controller.abort(), 30000); await fetch(webhook.url, { method: 'POST', headers, body, signal: controller.signal }); clearTimeout(timeoutId); Si se mantiene BullMQ para el disparo asíncrono (recomendado en punto 1.4), este worker corre en Node.js y sí puede usar fetch nativo sin problemas.

📌 ACCIÓN REQUERIDA: Sincronización del disparo de Pusher desde la API Pública
Cuando un sistema externo (POS, Zapier) crea una reserva via POST /api/v1/bookings, el Dashboard del Módulo 3 NO se enterará en tiempo real a menos que se emita el evento de Pusher. Después de que el webhook de Cal.com confirme la reserva (BOOKING_CREATED), el handler del webhook debe emitir: await pusher.trigger(`private-restaurant-${restaurantId}`, 'new-booking', {   ...bookingData,   source: 'api',  // distinguir visualmente en el dashboard }); Sin esto, el personal del restaurante no verá las reservas del POS en tiempo real en el Dashboard hasta recargar la página.
🐛 BUG: Los headers del request son de solo lectura en Next.js App Router
En el middleware, el documento intenta pasar el restaurantId a la API Route modificando los headers: request.headers.set('x-restaurant-id', keyRecord.restaurantId); PROBLEMA: En Next.js App Router, los headers del request son inmutables (ReadonlyHeaders). Esta línea lanza un TypeError en tiempo de ejecución. CORRECCIÓN: Usar NextResponse con headers adicionales o pasar el dato via un header en la respuesta que se reenvía: // CORRECTO: crear nuevo objeto de headers const requestHeaders = new Headers(request.headers); requestHeaders.set('x-restaurant-id', keyRecord.restaurantId); requestHeaders.set('x-restaurant-plan', keyRecord.restaurant.plan); return NextResponse.next({ request: { headers: requestHeaders } });
🐛 BUG: La firma del webhook omite el prefijo estándar de la industria
En lib/webhooks.ts, la firma se genera y envía así: const signature = crypto.createHmac('sha256', webhook.secret).update(body).digest('hex'); headers: { 'X-Webhook-Signature': signature } PROBLEMA: El estándar de la industria (Stripe, GitHub, Shopify, Cal.com) antepone 'sha256=' a la firma para identificar el algoritmo. Sin este prefijo, los desarrolladores que integren la API no podrán verificar la firma usando los snippets estándar de cualquier lenguaje. CORRECCIÓN: 'X-Webhook-Signature': `sha256=${signature}`, Actualizar la documentación y los snippets del SDK para reflejar el prefijo. Los desarrolladores que ya estén usando el formato sin prefijo necesitarán migrar, por lo que es crítico corregir esto antes del primer lanzamiento público.
🐛 BUG DE SEGURIDAD: Cualquier API Key puede consultar la disponibilidad de cualquier restaurante
El endpoint GET /api/v1/availability consulta Cal.com sin verificar que el eventTypeId o el restaurantId solicitado pertenece al restaurante autenticado. ESCENARIO: El restaurante A conoce el eventTypeId del restaurante B (visible en el HTML de la página pública) y puede consultar su disponibilidad con su propio API Key. Si bien esto solo expone información de disponibilidad (no datos sensibles), viola el principio de aislamiento de tenants y puede usarse para espionaje competitivo. SOLUCIÓN: Verificar siempre que el eventTypeId pertenece al restaurantId autenticado: const eventType = await db.tableType.findFirst({   where: { calEventTypeId: params.eventTypeId, restaurantId: req.headers.get('x-restaurant-id') } }); if (!eventType) return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
🐛 VULNERABILIDAD: Un cliente externo puede mover reservas entre restaurantes
El endpoint PUT /api/v1/bookings/:id actualiza el estado de una reserva (check-in, no-show, cancelado). Sin embargo, el documento no verifica que la reserva a actualizar pertenezca al restaurante autenticado. Un atacante con un API Key válido (de cualquier restaurante) podría hacer check-in o cancelar reservas de otros restaurantes si conoce el ID de la reserva. SOLUCIÓN OBLIGATORIA: Igual que en el Módulo 4 (cancelBooking), verificar ownership antes de cualquier mutación: const reservation = await db.reservation.findUnique({ where: { id: params.id } }); if (!reservation || reservation.restaurantId !== restaurantId) {   return NextResponse.json({ error: 'Not Found' }, { status: 404 }); } Retornar 404 (no 403) para no confirmar si el recurso existe en otro restaurante.
🐛 BUG: createBooking() en el SDK no tiene try/catch ni manejo de errores HTTP
El snippet del SDK en sdk/src/client.ts llama a fetch sin verificar response.ok: async createBooking(bookingData: BookingPayload) {   const response = await fetch(`${this.baseUrl}/bookings`, { ... });   return response.json();  // ← si es 400/429/500, retorna el objeto de error sin lanzar excepción } PROBLEMA: Si el servidor retorna 429 (rate limit) o 400 (datos inválidos), el SDK retorna el objeto de error JSON silenciosamente. El desarrollador que usa el SDK no sabe que falló a menos que inspeccione el resultado. CORRECCIÓN estándar para SDKs: const response = await fetch(...); if (!response.ok) {   const errorBody = await response.json().catch(() => ({}));   throw new AntigravityApiError(response.status, errorBody.error ?? 'Unknown error'); } return response.json();
⚠️ PROBLEMA CRÍTICO DE SEGURIDAD: El hash del API Key no está contemplado
El modelo ApiKey almacena la clave como string en texto plano en PostgreSQL. Si la base de datos es comprometida, todos los API Keys quedan expuestos. El estándar de la industria (Stripe, GitHub) es almacenar ÚNICAMENTE el hash SHA-256 de la clave, nunca la clave en sí. La clave en texto plano solo se muestra UNA vez al crearse y nunca se vuelve a mostrar. IMPLEMENTACIÓN CORRECTA: // Al crear el API Key: const rawKey = `res_live_${crypto.randomBytes(32).toString('hex')}`; const keyHash = crypto.createHash('sha256').update(rawKey).digest('hex'); await db.apiKey.create({ data: { keyHash, name, restaurantId, ... } }); // Devolver rawKey al usuario UNA sola vez — no se puede recuperar después  // Al validar en el middleware: const keyHash = crypto.createHash('sha256').update(apiKey).digest('hex'); const keyRecord = await prisma.apiKey.findUnique({ where: { keyHash } }); El campo clave en el schema debe renombrarse a keyHash y marcarse con @unique.
⚠️ FUNCIONALIDAD FALTANTE: El documento menciona scopes pero no los implementa
El documento describe 'Opción de generar múltiples claves con permisos específicos (solo lectura, solo escritura)' pero el schema de ApiKey no incluye un campo de scopes/permisos. Consecuencia: Todas las claves tienen acceso total. Una clave dada a un partner de analytics (solo lectura) podría usarse para crear o cancelar reservas. IMPLEMENTACIÓN RECOMENDADA: Agregar campo scopes al schema: model ApiKey {   scopes String[]  // ['bookings:read', 'bookings:write', 'availability:read', 'tables:write'] } En cada endpoint, verificar el scope requerido después de autenticar: const requiredScope = 'bookings:write'; if (!keyRecord.scopes.includes(requiredScope)) {   return NextResponse.json({ error: 'Insufficient permissions' }, { status: 403 }); }
Scope | Endpoints Autorizados | Caso de Uso
bookings:read | GET /api/v1/bookings | Sistema de analytics o BI externo
bookings:write | POST, PUT, DELETE /api/v1/bookings | POS del restaurante que crea reservas
availability:read | GET /api/v1/availability | App móvil del restaurante o Google Maps
tables:write | PUT /api/v1/tables/:id | Sistema de floor plan externo
customers:read | GET /api/v1/customers | CRM externo o herramienta de marketing
webhooks:manage | Dashboard de configuración de webhooks | Solo el dueño del restaurante
⚠️ CONSIDERACIÓN DE SEGURIDAD: /api/docs expone la estructura interna sin auth
El middleware excluye explícitamente /api/docs de la autenticación. Esto permite que cualquier persona en internet acceda a la documentación completa de la API. Si bien la documentación pública puede ser un objetivo (para una plataforma open-partner como Stripe), para el MVP de Antigravity esto expone: • La estructura de datos interna (campos, tipos, validaciones). • Los nombres de los endpoints y los scopes de permisos. • Posibles campos internos que no deberían ser públicos (ej. internalNotes en Customer). RECOMENDACIÓN para MVP: Proteger /api/docs con autenticación de Clerk. Solo usuarios con restaurante activo pueden ver la documentación. RECOMENDACIÓN para Fase 2 (plataforma abierta): Crear dos versiones de la spec — una pública (solo endpoints de disponibilidad y reservas) y una privada (todos los endpoints).
⚠️ PROBLEMA: Un evento de Cal.com puede generar dos webhooks salientes al sistema externo
Cuando Cal.com dispara el webhook BOOKING_CREATED, el handler de Antigravity:   1. Escribe en la tabla Reservation de PostgreSQL.   2. Debe disparar el webhook saliente al POS o Zapier del restaurante. Si el handler falla después del punto 1 pero antes del punto 2 y se reintenta (BullMQ), la reserva ya está en BD pero el webhook vuelve a intentarse. El POS del restaurante recibirá booking.created dos veces para la misma reserva. SOLUCIÓN: Implementar idempotency key en la tabla WebhookDelivery: model WebhookDelivery {   idempotencyKey String  @unique  // = `${webhookId}:${event}:${bookingId}`   status         String  // 'success' | 'failed' | 'pending' } Antes de disparar, verificar si ya existe un WebhookDelivery con ese idempotencyKey y status='success'.
⚠️ GAP: Un endpoint de webhook caído indefinidamente seguirá acumulando reintentos
El documento define reintentos a los 5 min, 30 min y 4 horas. Pero no define qué ocurre después del tercer intento fallido. ESCENARIO: El servidor del restaurante lleva 3 días caído. BullMQ sigue intentando enviar cada 4 horas, acumulando jobs fallidos en la queue. Esto genera: • Acumulación de jobs en Redis que consumen memoria. • Ruido en los logs que oculta errores reales. • Si el servidor vuelve después de 3 días, recibe 100+ webhooks de golpe. SOLUCIÓN: Política de desactivación automática: // Después de 5 fallos consecutivos: // 1. Marcar el webhook como status='suspended' en BD // 2. Notificar al dueño del restaurante por email: //    'Tu webhook a https://... ha fallado 5 veces. Lo hemos pausado.' // 3. Mostrar alerta en el dashboard con botón 'Reactivar' Definir en la tabla Webhook los campos: consecutiveFailures (Int, default 0) y status ('active' | 'suspended' | 'disabled').
⚠️ RIESGO: Los logs de webhook almacenan el payload completo incluyendo PII del cliente
La tabla WebhookDelivery almacena el payload completo del evento (booking.created incluye nombre, email y teléfono del cliente). Cualquier miembro del staff con acceso al panel puede ver estos datos en el historial de entregas. En Colombia, la Ley 1581 (Habeas Data) exige minimización de datos. El historial de logs no debería almacenar datos personales completos. SOLUCIÓN: Anonimizar el PII en el payload almacenado: const sanitizedPayload = {   ...payload,   customerName: payload.customerName?.slice(0, 1) + '***',   customerEmail: maskEmail(payload.customerEmail),   customerPhone: payload.customerPhone?.slice(-4).padStart(10, '*'), }; await db.webhookDelivery.create({ data: { ..., payload: sanitizedPayload } }); El payload completo ya está en la tabla Reservation; los logs solo necesitan el ID de la reserva para debugging.
⚠️ GAP CRÍTICO: Google Reserve requiere un proceso de aprobación de hasta 30 días
El documento menciona 'Requiere validación de Google (Google Business Profile API)' sin detallar el proceso real. Google Reserve NO es una integración que se activa con un API Key. Requiere:   1. Registro como partner de Reserve with Google (proceso manual, formulario).   2. Integración con la especificación de Google Reserve feeds (formato específico de XML o API).   3. Revisión y aprobación por parte de Google (puede tomar 2-8 semanas).   4. Cumplimiento continuo de las políticas de Google Business. RECOMENDACIÓN: No incluir Google Reserve en el MVP. Documentar como 'Próximamente' en el dashboard. Iniciar el proceso de registro con Google en paralelo al desarrollo para tenerlo listo en Fase 2. ALTERNATIVA más rápida para Fase 1: Implementar Google Business Profile API (no Reserve) que permite actualizar horarios, responder reseñas y ver métricas. La API de Business Profile SÍ tiene acceso inmediato con OAuth.
⚠️ GAP: La integración de Zapier está incompleta para publicarse en el marketplace
El documento describe la integración Zapier desde el punto de vista del webhook saliente (Antigravity notifica a Zapier). Pero para aparecer en el marketplace de Zapier como una app oficial, se requiere:   1. Crear una Zapier CLI App con autenticación OAuth 2.0 (no API Key).   2. Definir Triggers (eventos que Zapier puede escuchar) Y Actions (cosas que Zapier puede hacer en Antigravity).   3. Publicación en el marketplace de Zapier (revisión manual, ~2-4 semanas). RECOMENDACIÓN MVP: Implementar integración via webhook genérico (el sistema de webhooks del punto 2 ya cubre esto sin necesidad de una app de Zapier oficial). Documentar en la guía de inicio rápido cómo configurar Zapier usando el webhook genérico de Antigravity. La app oficial de Zapier es una mejora de Fase 2 que requiere un proceso de publicación formal.
⚠️ GAP: La integración con Vendty/SoftRestaurant requiere mapeo de campos específico
El documento menciona 'Mapeo de campos (Nombre cliente -> Orden POS)' sin definir el mapeo completo. Este mapeo es fundamental para que la integración sea útil y varía por POS. CAMPOS MÍNIMOS que debe soportar el endpoint /api/webhooks/incoming para un POS:  El mapeo de mesas (tableTypeId → número de mesa en el POS) requiere una tabla de configuración por restaurante: PosTableMapping { antigravityTableId, posTableNumber }. Sin esta tabla, las reservas aparecerán en el POS sin asignación de mesa.
⚠️ RIESGO: La spec de OpenAPI puede desincronizarse del código real
El documento propone usar next-swagger-doc donde los desarrolladores escriben los comentarios JSDoc manualmente en cada endpoint. Esto genera un riesgo crítico a largo plazo: cuando un endpoint cambia su schema (nuevo campo, campo eliminado), el desarrollador puede olvidar actualizar el comentario JSDoc. CONSECUENCIA: Un desarrollador externo leerá en la doc que la respuesta incluye el campo depositAmount pero el endpoint ya no lo devuelve → integración rota. SOLUCIÓN: Generar la documentación desde la validación Zod (que ya es la fuente de verdad del código): import { generateOpenApiDocument } from 'zod-to-openapi'; Librería recomendada: @asteasolutions/zod-to-openapi. Define los schemas Zod una sola vez y genera tanto la validación del endpoint como la spec OpenAPI automáticamente. Con este enfoque, si el schema Zod cambia → la documentación cambia automáticamente → nunca hay desincronización.
⚠️ GAP: El entorno de prueba puede contaminar datos reales del restaurante
El documento menciona un 'Sandbox para probar sin afectar datos reales' pero no especifica cómo se aíslan los datos de prueba. Si el sandbox usa el mismo restaurantId que el entorno live, una reserva de prueba creada desde Swagger puede:   • Aparecer en el dashboard del restaurante como una reserva real.   • Bloquear slots en Cal.com (si el flujo correcto pasa por Cal.com).   • Disparar emails de confirmación al cliente ficticio. SOLUCIÓN: Las claves de tipo res_test_... deben: 1. Usar un restaurante de prueba separado en BD (restaurantId de sandbox) 2. Llamar a Cal.com en modo de prueba (sandbox environment de Cal.com si existe) 3. No disparar emails ni WhatsApp reales (usar Resend en modo sandbox) 4. Limpiar automáticamente las reservas de prueba después de 24 horas (cron job) Agregar campo environment a la tabla Restaurant: 'production' | 'sandbox'.
⚠️ RIESGO A LARGO PLAZO: /api/v1 sin política de deprecación genera deuda técnica
El documento define la versión v1 en la URL (/api/v1/bookings) pero no documenta la política de versionado: cuándo se crea una v2, cuánto tiempo vive la v1, cómo se comunica la deprecación a los integradores. Sin esta política, cualquier cambio breaking (renombrar un campo, cambiar el formato de una fecha) requiere:   A) Mantener compatibilidad hacia atrás indefinidamente (acumula deuda técnica).   B) Romper la integración de todos los POS y partners sin previo aviso. POLÍTICA MÍNIMA RECOMENDADA (definir ahora, antes de lanzar): • Una versión se depreca con mínimo 6 meses de aviso previo. • El aviso se envía por email a todos los restaurantes con claves activas. • Header de respuesta: 'Deprecation: true' + 'Sunset: <fecha>' cuando un endpoint está deprecated. • Mantener v1 activa 12 meses después del lanzamiento de v2.
⚠️ RIESGO DE UX: Un usuario puede confundir una clave de test con una de producción
El dashboard muestra las API Keys en una tabla sin diferenciar visualmente entre res_live_... y res_test_.... En producción, un restaurante puede copiar una clave de test a su POS real y no entender por qué las reservas no aparecen en Cal.com. SOLUCIÓN: En el componente ApiKeyManager.tsx, mostrar badges visuales:   • Clave LIVE: badge verde 'Producción'. Fondo de la fila blanco.   • Clave TEST: badge amarillo 'Pruebas'. Fondo de la fila amarillo suave.   • Agregar tooltip: 'Las claves de prueba no crean reservas reales ni envían notificaciones.'   • En el formulario de creación, forzar al usuario a elegir explícitamente 'Producción' o 'Pruebas'.

⚠️ REQUISITO LEGAL: Ley 1581 exige informar al usuario qué datos se transfieren a terceros
El grid de apps (Google, Slack, Zapier, POS) muestra un toggle de conexión pero no informa al restaurante (ni a sus clientes) qué datos personales de clientes se transfieren a cada sistema externo. REQUERIMIENTO OBLIGATORIO: Antes de activar cualquier integración, mostrar un modal de consentimiento informado que especifique:   • Qué datos se transfieren (nombre, email, teléfono del cliente).   • Con qué sistema se comparten (ej. 'Slack recibirá el nombre del cliente VIP').   • Dónde se almacenan esos datos (país/región del servidor del tercero).   • Cómo desactivar la transferencia (toggle en el panel). Registrar en la tabla ConsentRecord del restaurante la activación de cada integración con timestamp y versión de la política.
