# CONTEXTO DEL PROYECTO
Estás construyendo una plataforma SaaS de reservas para
restaurantes (similar a OpenTable/Precompra pero mejorada).
--- Page 4 ---
El sistema usará Cal.com como backend de reservas
(self-hosted).
Esta es la PRIMERA PARTE: El sistema de
onboarding/configuración self-service para restaurantes.
STACK TECNOLÓGICO:
- Frontend: Next.js 14+ (App Router), TypeScript, TailwindCSS
- UI Components: shadcn/ui, Radix UI
- Drag & Drop: dnd-kit o react-beautiful-dnd
- Image Upload: UploadThing o Cloudinary
- Color Picker: react-colorful o @uiw/react-color
- QR Generation: qrcode.react o node-qrcode
- State Management: Zustand o React Context
- Forms: React Hook Form + Zod validation
- Backend: Next.js API Routes + Prisma ORM
- Database: PostgreSQL
- Auth: NextAuth.js o Clerk
---
## CARACTERÍSTICA 1: SETUP WIZARD DE 5 PASOS
### REQUISITOS FUNCIONALES:
Crea un wizard multi-paso progresivo que guíe al restaurante
en su configuración inicial.
**ESTRUCTURA DEL WIZARD:**
Paso 1: Información Básica
- Campo: Nombre del restaurante (texto, requerido, max 100
chars)
- Campo: NIT/Documento de identidad (texto, requerido)
--- Page 5 ---
- Campo: Tipo de establecimiento (select: Restaurante, Bar,
Cafetería, Food Truck, Otro)
- Campo: Descripción corta (textarea, max 250 chars)
- Campo: Dirección completa (texto con autocompletado de
Google Places API)
- Campo: Ciudad (select dinámico basado en Google Places)
- Campo: Teléfono principal (input tipo tel con validación de
formato)
- Campo: Email de contacto (input email, validación)
- Campo: Sitio web (input url, opcional)
- Campo: Redes sociales (inputs opcionales: Instagram,
Facebook, TikTok)
Paso 2: Horarios de Atención
- Componente: Grid de 7 días (Lunes a Domingo)
- Por cada día:
* Checkbox "Abierto este día"
* Si está abierto:
- Selector de hora de apertura (time picker HH:mm)
- Selector de hora de cierre (time picker HH:mm)
- Botón "Agregar turno" (para restaurantes con almuerzo y
cena)
- Múltiples turnos soportados (ej: 12:00-15:00,
19:00-23:00)
- Toggle: "Horarios especiales para festivos"
- Selector de zona horaria (default: America/Bogota)
Paso 3: Configuración de Capacidad y Mesas
- Campo: Capacidad total máxima (número, min 1)
- Campo: Duración promedio de reserva (select: 30min, 45min,
1h, 1h30m, 2h, 2h30m, 3h)
- Campo: Tiempo entre reservas (buffer time, select: 0min,
15min, 30min)
- Sección: Configuración de mesas
--- Page 6 ---
* Botón "Agregar tipo de mesa"
* Por cada tipo de mesa:
- Nombre (ej: "Mesa para 2", "Mesa VIP", "Barra")
- Capacidad mínima (número)
- Capacidad máxima (número)
- Cantidad de mesas disponibles (número)
- Upload de imagen opcional (foto de referencia)
- Toggle: "Permitir reservas para grupos grandes (+10
personas)"
- Si es sí: Campo "Capacidad máxima por reserva" (default 20)
Paso 4: Configuración de Pagos y Depósitos
- Toggle: "Requerir depósito para reservar"
- Si es sí:
* Radio buttons: Tipo de depósito
- "Porcentaje del estimado" (input número, min 10, max
100, default 50)
- "Monto fijo" (input número con selector de moneda)
- "Solo garantía de tarjeta" (sin cobro, solo retención)
* Toggle: "Cobrar automáticamente por no-show"
* Si es sí: Input "Horas después para cobrar" (default 2)
* Toggle: "Permitir reembolsos"
* Si es sí: Input "Horas antes para reembolso completo"
(default 24)
- Selector: Pasarela de pagos (Wompi, PayU, Stripe, ePayco)
- Campos dinámicos según pasarela seleccionada:
* Wompi: Public Key, Secret Key
* Stripe: Publishable Key, Secret Key
* etc.
- Toggle: "Aceptar efectivo en el local" (default sí)
Paso 5: Revisión y Publicación
- Vista de resumen de todos los datos ingresados
- Checklist de completitud:
--- Page 7 ---
* ✓ Información básica completada
* ✓ Horarios configurados
* ✓ Al menos 1 tipo de mesa creado
* ✓ Método de pago configurado (si requiere depósito)
- Toggle: "Acepto términos y condiciones" (requerido)
- Toggle: "Acepto política de privacidad" (requerido)
- Botón principal: "Publicar mi restaurante" (deshabilitado
hasta que todo esté completo)
- Botón secundario: "Guardar como borrador"

async function getCalComAvailability(

): Promise<AvailabilityResponse> {

async function updateCustomerSegments(restaurantId: string) {
const customers = await prisma.customer.findMany({
where: { restaurantId },
include: { reservations: true },
});
for (const customer of customers) {
let segment = 'new';
const daysSinceLastVisit = customer.lastVisitAt
? differenceInDays(new Date(), customer.lastVisitAt)
: Infinity;
if (customer.totalVisits >= 10 || customer.totalSpent >=
500000) {
segment = 'vip';
} else if (customer.totalVisits >= 3) {
segment = 'regular';
} else if (customer.totalVisits >= 1) {
segment = 'new';
}
// Check for at-risk
if (segment === 'regular' && daysSinceLastVisit > 90) {
segment = 'at-risk';
}
// Check for inactive
if (daysSinceLastVisit > 180) {
segment = 'inactive';
}
await prisma.customer.update({
where: { id: customer.id },
data: { segment },
});
}
}

function generateInsights(metrics: AnalyticsMetrics):
Insight[] {
const insights: Insight[] = [];

if (metrics.reservationGrowth > 10) {
insights.push({
type: 'positive',
📈
message: ` Tus reservas aumentaron
${metrics.reservationGrowth}% este período`,
metric: 'reservationGrowth',
value: metrics.reservationGrowth,
});
}
// No-show rate
if (metrics.noShowRate > 10) {
insights.push({
type: 'warning',
⚠
message: ` Tu tasa de no-shows es
${metrics.noShowRate}% (promedio industry: 8%)`,
metric: 'noShowRate',
value: metrics.noShowRate,
});
--- Page 44 ---
}
// Popular day
const popularDay = metrics.dayOfWeekDistribution.sort((a, b)
=> b.count - a.count)[0];
insights.push({
type: 'suggestion',
💡
message: ` ${popularDay.day} es tu día más popular con
${popularDay.count} reservas`,
metric: 'popularDay',
value: popularDay.count,
});
// Source distribution
const topSource = metrics.reservationsBySource.sort((a, b)
=> b.count - a.count)[0];
insights.push({
type: 'suggestion',
📱
message: ` El ${Math.round(topSource.count /
metrics.totalReservations * 100)}% de tus reservas vienen por
${topSource.source}`,
metric: 'topSource',
value: topSource.count,
});
return insights;
}

const socket = io(process.env.NEXT_PUBLIC_WS_URL!);
socket.emit('join-conversation', conversationId);
socket.on('new-message', (message: Message) => {
setMessages(prev => [...prev, message]);

return () => {
socket.disconnect();

Estás construyendo un sistema de AGENTES DE IA UNIFICADOS que

export async function saveConversationHistory(
conversationId: string,
messages: Array<{ role: string; content: string }>
) {
const key = `conversation:${conversationId}:history`;
// Obtener historial existente
const existing = await
getConversationHistory(conversationId);
// Agregar nuevos mensajes
const updated = [...existing, ...messages].slice(-20); //
Mantener últimos 20
// Guardar con TTL de 24 horas
await redis.setex(key, 86400, JSON.stringify(updated));
}

string) {
await redis.del(`conversation:${conversationId}:history`);

export async function triggerWebhook(event: string, payload:

5.2 Flujo de una Reserva por WhatsApp
Cliente envía "Hola, quiero reservar una mesa para 4 personas este sábado".
WhatsApp API hace POST al webhook /api/webhooks/whatsapp.
El sistema identifica el restaurante por el número de WhatsApp Business.

GPT-4o responde con opciones de horario disponibles.
Cliente selecciona horario → agente llama a create_reservation().
Se crea reserva en Cal.com. Se guarda en PostgreSQL. Se envía email de confirmación.
WhatsApp responde: "Tu reserva está confirmada para el sábado 21 de junio a las 8pm..."
